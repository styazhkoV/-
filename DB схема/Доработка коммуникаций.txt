# BRD: Фиксация коммуникаций (Activity) при рассылках

---

## 1. Бизнес-потребность

На текущий момент CRM-система не фиксирует факт отправки рассылок клиентам. При отправке разовой или массовой рассылки запись в системе не создается. Это приводит к:
- Отсутствию истории коммуникаций с клиентом
- Невозможности аудита отправленных рассылок
- Дублированию коммуникаций (повторная отправка без контроля)
- Отсутствию данных о рассылках в отчетности

---

## 2. Scope (границы задачи)

### В scope:
- Сохранение Activity при разовой рассылке
- Сохранение Activity при массовой рассылке (на каждого получателя)
- Отображение Activity в детализации клиента (GET `/api/v1/kcell_client/{id}`)
- Отображение рассылок в отчетности
- Защита от дублирования записей
---

## 3. Функциональные требования

### FR-1. Запись Activity при разовой рассылке

**Описание:** После успешной отправки разовой рассылки клиенту система должна создать запись в таблице `client.activities`.

**Условия:**
- Запись создается ТОЛЬКО после успешной отправки
- При ошибке отправки запись НЕ создается
- Если запись с таким же `client_id + source_id + activity_type` уже существует — повторная запись не создается

**Данные записи:**

| Поле | Значение | Источник |
|---|---|---|
| `client_id` | ID клиента-получателя | Из параметров рассылки |
| `activity_type` | `"MAILING"` | Константа |
| `subject` | Тема рассылки | Из параметров рассылки |
| `description` | Текст сообщения | Из параметров рассылки |
| `source` | `"Kcell Mailing"` | Константа |
| `source_id` | ID рассылки | Из параметров рассылки |
| `created_by` | ID текущего пользователя | Из контекста авторизации |
| `created_at` | Текущая дата/время | Автоматически |

---

### FR-2. Запись Activity при массовой рассылке

**Описание:** При массовой рассылке система должна создать отдельную запись Activity для КАЖДОГО клиента-получателя.

**Условия:**
- Одна запись = один клиент
- Все записи привязаны к одному `source_id` (ID рассылки)
- Защита от дублей работает так же, как в FR-1

---

### FR-3. Отображение Activity в детализации клиента

**Описание:** API детализации клиента (`GET /api/v1/kcell_client/{id}?clientType={type}`) должен возвращать список Activity данного клиента.

**Ответ API — секция `activities`:**

```json
{
  "activities": [
    {
      "activityId": "123",
      "activityType": "MAILING",
      "createdAt": "2025-03-26T10:30:00",
      "createdBy": "Иванов И.И.",
      "description": "Текст рассылки..."
    }
  ]
}
```

**Условия:**
- Возвращаются все Activity клиента (не только MAILING)
- Сортировка: по `created_at` DESC (новые сверху)
- Данные из БД (реальные, не моковые)

---

### FR-4. Рассылки в отчетности

**Описание:** В модуле отчетов должна быть возможность получить данные по отправленным рассылкам.

**Источник данных:** таблица `client.activities` с фильтром `activity_type = 'MAILING'`

**Поля отчета:**
- Клиент (имя / наименование компании)
- Тема рассылки
- Дата отправки
- Отправитель (сотрудник)
- ID рассылки
