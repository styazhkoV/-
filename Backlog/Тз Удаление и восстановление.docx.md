**3\. Функциональные требования к интерфейсу**

**3.1. Процесс удаления**

1. **Кнопка «Удалить»:** Доступна только пользователям с соответствующими правами.

   * Пока можно оставить только для роли ROLE\_ADMIN, в будущем будем конкретизировать какие роли могут. 

2. В списке клиентов добавить колонку (справа) и возможность удалить.

3. **Модальное окно:** \* Если валидация прошла \- вопрос: *«Вы уверены?»*.

   * Если валидация не прошла \- список причин: *«Удаление невозможно: \[Причина 1\], \[Причина 2\]»*.

4. **Результат:** После удаления клиент исчезает из общего списка и поиска.

**3.2. Процесс восстановления** 

1. **Реестр удаленных записей:** В общем списке клиентов добавить фильтр «Удаленные клиенты».

2. **Табличное представление:**

   * В списке удаленных клиентов добавить колонку «Действия» (справа).

   * В этой колонке для каждой строки должна быть активна кнопка (или иконка) «Восстановить» (например, иконка Undo или RotateRight).

3. **Логика восстановления:**

   * При нажатии на кнопку во фронтенде вызывается подтверждающее модальное окно: *«Восстановить карточку клиента \[Имя\]?»*.

   * После подтверждения флаг is\_deleted меняется на false.

   * Запись исчезает из списка «Удаленные» и снова становится доступной в основном реестре «Активные».

**5\. Критерии приемки (Acceptance Criteria)**

1. \[ \] Система не дает удалить клиента с балансом $\\neq 0$;

2. \[ \] Система не дает удалить клиента, если по нему открыт тикет в саппорте и иные активности;

3. \[ \] После удаления лиды, но теряют связь с данной карточкой;

4. \[ \] Удаленный клиент доступен для просмотра через специальный фильтр «Удаленные» у сотрудника с соответсвующими правами;

5. \[ \] В таблице удаленных клиентов крайняя правая колонка содержит функциональную кнопку восстановления;

6. \[ \] При нажатии кнопки восстановления в таблице статус записи в БД меняется мгновенно, и строка удаляется из текущего экранного представления (архива).
