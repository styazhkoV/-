 1. Тестирование справочников и связей (DTO & Entities)

**Цель:** Проверить корректность маппинга данных и отображения человекочитаемых значений вместо ID.

* **1.1. Настройка связей `ref_` в Client, ClientB2C/B2B:**
  * **Шаг:** Через БД или API проверить наличие связей сущностей со справочниками (гражданство, сегмент, тип клиента и т.д.).
  * **Ожидаемый результат:** В БД установлены Foreign Keys; при запросе объекта Client возвращаются корректно связанные объекты справочников.
* **1.2. Маппинг DTO для детализации:**
  * **Шаг:** Вызвать GET-метод детализации клиента (GET `/clients/{id}`).
  * **Ожидаемый результат:** В ответе API вместо `id_marital_status: 1` приходят текстовые значения (напр., `marital_status: "Женат/Замужем"`), соответствующие маппингу DTO.

 2. Управление данными клиента (Fixes & New API)

**Цель:** Проверить стабильность сохранения данных и поиск.

* **2.1. Добавление контактных и доверенных лиц (POST):**
  * **Шаг:** Отправить POST запрос на эндпоинт записи контактных лиц с валидным JSON.
  * **Ожидаемый результат:** Код 201 Created. Запись появилась в БД и корректно привязана к ID основного клиента.
  * **Негативный тест:** Отправка пустых обязательных полей или некорректного формата телефона/ИИН.
* **2.2. Ошибка Duplicate Key (Семейное положение и кодовое слово):**
  * **Шаг:** Попробовать сохранить/обновить семейное положение или кодовое слово для одного и того же клиента несколько раз подряд.
  * **Ожидаемый результат:** Ошибка `duplicate key value violates unique constraint` не возникает. Данные успешно перезаписываются (UPSERT).
* **2.3. Сохранение адресов и Email (Доп. информация):**
  * **Шаг:** В интерфейсе/API обновить данные во вкладке «Дополнительная информация».
  * **Ожидаемый результат:** Данные сохраняются без потерь, формат email проходит валидацию, адреса корректно пишутся в связанные таблицы.
* **2.4. Поиск по ИИН/БИН:**
  * **Шаг:** Выполнить поиск клиента по полному и частичному ИИН/БИН.
  * **Ожидаемый результат:** Клиент находится мгновенно. В списке (grid) ИИН/БИН отображается без обрезания цифр и масок (если не предусмотрено иное).

| **Шаг** | **Действие**                                                              | **Ожидаемый результат**                                                                  | **Технический нюанс**                                                                                                                                 |
| ---------------- | --------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1                | **POST**данных формы «Контактные лица».                | Статус 201. Объект успешно создан.                                                      | Проверить в DBeaver таблицу `client_contacts`:`client_id`должен быть корректно привязан.                                    |
| 2                | **Повторное сохранение**семейного положения. | Статус 200/204. Ошибка `duplicate key`не возникает.                                     | В DBeaver убедиться, что запись обновилась, а не создалась новая (если логика — 1 клиент = 1 статус).  |
| 3                | **Сохранение адресов/email**в Доп. информации.     | Все поля сохраняются и возвращаются при повторном GET-запросе. | Проверить, что email не обрезается и записывается в соответствующее поле (без смещения колонок). |

 3. Campaign Management & CRM

**Цель:** Проверка UI-логики и прав доступа.

* **3.1. Наполнение селектов и сортировка:**
  * **Шаг:** Открыть раздел Campaign Management. Проверить выпадающие списки (Selects). Проверить сортировку по ключевым столбцам.
  * **Ожидаемый результат:** Селекты наполнены данными из справочников. Сортировка работает корректно (ASC/DESC) без ошибок бэкенда.
* **3.2. Ошибка 401 Unauthorized (Создание пользователя CRM):**
  * **Шаг:** Попытаться создать пользователя типа CRM под учетной записью с соответствующими правами.
  * **Ожидаемый результат:** Пользователь создается успешно. Ошибка 401 (неавторизован) не воспроизводится при наличии валидного токена.

 Сценарий 3: Поиск и безопасность

**Инструменты:** Postman.

* **Поиск по ИИН/БИН:**
  * Выполнить поиск по существующему ИИН.
  * **Ожидаемый результат:** В ответе массив объектов, где ИИН отображается полностью (без экспоненциальной записи типа `2.1E+11`).
* **Исправление 401 Unauthorized:**
  * Создать пользователя типа CRM с валидным токеном.
  * **Ожидаемый результат:** Статус 201.
  * **Негативный тест:** Умышленно испортить токен — должна вернуться 401. Убедиться, что ошибка 401 не возникает *при легитимном* запросе.

 Сценарий 4: Регрессия и системные проверки

**Инструменты:** DBeaver, Postman.

1. **Проверка связей (Foreign Keys):** В DBeaver через вкладку "ER Diagram" или "Constraints" убедиться, что для сущностей `Client`, `ClientB2B/B2C` настроены связи с таблицами `ref_`.
2. **Notification Service (Prod Check):**
   * Отправить триггерное событие (например, изменение статуса клиента).
   * Проверить, ушло ли уведомление (если есть доступ к логам или таблице `notifications_log`).
3. **Сортировка в Campaign Management:**
   * Добавить в запрос параметры `sort=name,asc` или `sort=date,desc`.
   * Убедиться, что порядок элементов в JSON меняется корректно.
